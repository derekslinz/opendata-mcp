# ADR 0002 — Autogen Plugin Security Posture

**Status**: Proposed
**Date**: 2026-05-17
**Supersedes**: none
**Superseded by**: none

## Context

`opendata-create-plugin` lets an LLM author scaffold a brand-new
provider module at runtime: take a YAML spec (produced by
`opendata-draft-spec` or hand-written), run the generator, import the
module, register the `ProviderEntry`, and merge its tools into the live
catalog. This is the "v1.3 agent-driven generation" capability shipped
in the v1.3 release — currently the meta server's most powerful tool.

It is also currently the meta server's largest attack surface. A spec
that the LLM accepts (or that a low-trust user provides) becomes a live
module that:

- Runs in the same Python process as the rest of the server.
- Inherits the kernel's network egress (any URL the kernel can reach,
  the plugin can reach).
- Persists across the rest of the server's lifetime once registered.
- Survives across MCP sessions if the file ends up on disk.

There is no current isolation. There is no current allow-list on
`base_url`. There is no current trust split between "ephemeral
sandbox for the session" and "promoted plugin everyone uses." There
is no current content-type sniffing on the plugin's outbound
responses — a plugin that returns `text/html` looking like JSON would
silently pass through serialization.

This ADR codifies the security posture we want before the next major
bump to `opendata-create-plugin` and before any hosted deployment
goes live.

## Decision

Four principles, listed in increasing implementation cost:

### 1. Per-session ephemeral plugins by default

`opendata-create-plugin` creates a plugin that is alive for the
**duration of the MCP session only**. The plugin module is loaded
into `sys.modules` and merged into `TOOLS` / `TOOLS_HANDLERS`, but
**not** written to `meta_data_mcp/providers/` on disk by default.
When the session ends, the plugin is gone.

This is the "spike-test the API for one conversation, throw away the
adapter" model. It costs nothing to the user (the LLM can recreate
the spec next session) and removes the persistence vector entirely.

### 2. Persistent plugins require explicit promotion

A new tool `opendata-promote-plugin(plugin_id)` writes the
session-ephemeral plugin's module file to
`meta_data_mcp/providers/<id>.py`. Promotion is the explicit moment
where a plugin transitions from "in this conversation only" to "in
the next process boot too."

Promotion requires either:
- A scoped bearer token with `admin` scope (see the planned OAuth2 /
  scoped-token work in the roadmap), or
- The local stdio transport (process-local trust boundary, no
  external attacker)

In the absence of scoped tokens (current v2.x state), promotion is
gated on the stdio transport for the development workflow and
explicitly denied on SSE.

### 3. `base_url` allow-list option

Operators can set `META_DATA_MCP_AUTOGEN_BASE_URL_ALLOWLIST` to a
comma-separated list of URL prefixes (`https://api.example.com,`
`https://data.gov`). When set, `opendata-create-plugin` rejects any
spec whose `base_url` doesn't start with one of the allowed prefixes.

Default unset = no allow-list = current behavior (any URL allowed).
Setting the env var is what a hosted deployment does to lock down
the egress surface.

### 4. Plugin-output content-type sniffing

The kernel's `http_get` and `http_post` already cache responses with
their `Content-Type` header. We extend the contract: when a plugin
hands a response to a size-bounded serializer
(`serialize_for_llm` / `to_records_text` / etc.), the serializer
**checks** that the response's declared `Content-Type` is in the
expected family for that shape. Mismatches log a `WARNING` and
include a `provenance.warnings` field in the tool result.

This catches the case where an upstream API returns an error HTML
page with a 200 status code — a class of bug that today is silent.

## Consequences

### Positive

- Ephemeral-by-default removes the persistence vector for malicious
  or buggy autogenerated plugins.
- Promotion as an explicit action makes "this plugin is now real"
  visible in logs and reviewable by an operator.
- The allow-list gives hosted deployments a single env var to bound
  the egress surface — no per-spec policy needed.
- Content-type sniffing surfaces an entire class of silent failures.

### Negative

- "Recreate from spec each session" is friction for power users who
  routinely re-author the same plugin. Mitigation: the LLM can keep
  the spec in memory and resubmit; spec files are 30-line YAML.
- Promotion gating on scoped tokens means the OAuth2 work
  (roadmap v2.3) is a hard prerequisite for safe SSE promotion.
  Until then, SSE deployments cannot promote.
- The allow-list adds an env var to the operations surface. Default
  unset preserves backwards compatibility.

### Neutral

- Subprocess / sandbox isolation (firejail, nsjail, V8-isolate-style)
  is **out of scope for this ADR** but tracked separately as P2 in
  the roadmap. ADR 0002 codifies the trust model; isolation is the
  enforcement mechanism that may layer on top later.

## Implementation order

The four principles are listed in implementation order:

1. **Ephemeral-by-default** — small `loader.py` change to make
   on-disk write a separate step. Adds `opendata-promote-plugin`.
2. **Promotion gating** — wired to transport detection until scoped
   tokens land (see "OAuth2 / scoped bearer tokens" in roadmap v2.3).
3. **Allow-list** — env var parsing + URL-prefix check in
   `_activate_provider`. Trivial change.
4. **Content-type sniffing** — serializer-side; one helper, one new
   `provenance.warnings` field on the result envelope.

Each step is shippable independently; no big-bang refactor.

## Revisit triggers

This ADR should be re-examined if any of the following happen:

- Public hosted SSE deployment goes live (the allow-list and
  promotion gating become operationally mandatory, not optional).
- An autogenerated plugin causes a security incident in any
  deployment (escalate isolation work from P2 to P0).
- The MCP protocol adds a session-scoped tool-catalog mechanism
  (would replace the "ephemeral default" implementation with a
  cleaner protocol-native equivalent).
